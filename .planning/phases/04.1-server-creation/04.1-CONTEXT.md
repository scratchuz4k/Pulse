# Phase 4.1: Server Creation - Context

**Gathered:** 2026-06-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Introduce the concept of a named "server" (organization) as a first-class entity. Users create servers and become their owner/admin; other users join via a static invite code. Rooms become server-scoped (each room belongs to one server). The server owner replaces the PULSE_ADMIN_USER_ID env-var stopgap for all admin-gated features. The client gains a server switcher (topbar dropdown), a Dashboard landing page with server creation/join UI, and server-scoped SignalR presence. Room.CreatedByUserId is removed.

</domain>

<decisions>
## Implementation Decisions

### Invite Code
- **D-01:** Each server has a **static permanent invite code** — a random string stored on the `Server` entity. It never expires. The owner can manually regenerate it from server settings.
- **D-02:** Joining is **immediate** — `POST /servers/join` with the invite code adds the caller as a `ServerMember` right away. No approval queue.
- **D-03:** The invite code is surfaced in **server settings** (copy button). Other users enter it in a **Join Server modal** (text input) accessed from the Dashboard.

### Admin Model Transition
- **D-04:** Server owner (`Server.OwnerId`) **fully replaces** `PULSE_ADMIN_USER_ID`. The env var is removed. All admin-gated hub methods (priority speaker, whisper group management) check `Server.OwnerId` instead.
- **D-05:** `Room.CreatedByUserId` is **removed** — it was explicitly a stopgap (see `Room.cs` comment). Admin identity now flows from the server owner.
- **D-06:** Creator is always the owner in v1 — no ownership transfer.

### Multi-Server UX
- **D-07:** Users can belong to **multiple servers** — create and/or join as many as they like. A `ServerMember` junction table tracks memberships.
- **D-08:** The server switcher is a **topbar dropdown** — shows the currently active server name; clicking opens a list of joined servers plus "Create server" and "Join server" options.
- **D-09:** The **Dashboard** (`/dashboard`) becomes the real landing page after login. If the user has no servers, it shows "Create a server" and "Join with invite code" CTAs. If they have servers, it shows their server list with quick-join links.
- **D-10:** After selecting a server, the user navigates to `/server` (the existing ServerTemplate) which loads rooms for that server.

### Room Scoping
- **D-11:** Rooms gain a **`ServerId` FK** (non-nullable). The dev DB is wiped on schema change (existing AppDbContext comment pattern — `EnsureCreated` dev strategy).
- **D-12:** Room listing uses a new **server-scoped endpoint**: `GET /servers/{id}/rooms`. The existing `GET /rooms` is either removed or restricted.
- **D-13:** Room creation is **`POST /servers/{id}/rooms`**, restricted to the **server owner only** (not any member).
- **D-14:** SignalR (PresenceHub) uses **server-scoped groups**. On `OnConnectedAsync`, the hub queries the user's `ServerMember` list and adds the connection to a SignalR group per server (`server-{serverId}`). All broadcasts (room joins, speaking, mute, whisper events) target the appropriate server group. The user receives presence updates from all their servers simultaneously.

### Claude's Discretion
- Exact `Server` entity fields beyond `Id`, `Name`, `OwnerId`, `InviteCode` (e.g., description, icon/color, createdAt).
- Naming convention for SignalR server groups (e.g., `server-{serverId}`).
- Whether `GET /servers/{id}` (fetch single server metadata) is needed in this phase or deferred.
- How the `activeServerId` is tracked in the client (Pinia store, router state, or both).
- Whether to expose a `GET /servers/me` endpoint listing the caller's servers, or drive the server list from a Pinia store populated at login.

</decisions>

## Existing Code Insights

### Reusable Assets
- `AppDbContext.cs` `EnsureCreated` dev pattern — wipe `pulse.db` and restart to apply schema. Follow this exact approach for the new Server/ServerMember tables.
- `AuthController.cs` `[Authorize]` + JWT claim extraction pattern — server endpoints use the same `User.FindFirst(ClaimTypes.NameIdentifier)` pattern to get `callerId`.
- `PresenceHub.cs` `ConcurrentDictionary` + `Groups.AddToGroupAsync` — server-group subscriptions follow the same pattern as existing room-group subscriptions.
- `RoomsController.cs` `DbUpdateException` HTTP 409 pattern — reuse for duplicate server names.
- `usePresence.ts` SignalR listener registration — add server-scoped event handlers here.
- `stores/room.ts` Pinia store — extend or companion-store for `activeServerId`.

### Established Patterns
- SignalR hub methods PascalCase, client events camelCase — follow for all new server/member hub methods.
- Admin check: compare caller's `userId` claim against a trusted identity before acting — now checks `server.OwnerId` instead of env var.
- Server entity persisted in DB (unlike ephemeral whisper groups) — use `DbSet<Server>` with EF Core.
- `RoomJoined` enriched-response pattern — server join response should include server metadata + room list.

### Integration Points
- `PresenceHub.OnConnectedAsync` — must be overridden (currently inherited implicit implementation) to subscribe the connection to per-server SignalR groups.
- `ServerTemplate.vue` topbar area — new server switcher dropdown added here.
- Router `beforeEach` guard — extend to check server membership and redirect to Dashboard if none.
- `DashBoard.vue` — becomes functional: fetches user's servers, shows create/join UI when empty.

</code_context>

<specifics>
## Specific Ideas

- Dashboard is the real post-login landing: shows "Create a server" + "Join with invite code" when user has no servers. Shows server cards when they do.
- Server switcher is a topbar dropdown (not a Discord-style left column).
- Invite code is plain text — user copies from server settings and shares out-of-band. No URL scheme.
- Room creation scoped to server owner only (not all members).
- On SignalR connect, hub subscribes the connection to ALL servers the user belongs to — user gets presence from all servers at once.

</specifics>

<deferred>
## Deferred Ideas

- Server icons/avatars — out of scope for v1; name + initials only.
- Ownership transfer — deferred; creator is always owner in v1.
- Invite link URL scheme (`pulse://join/<code>`) — deferred; plain code copy/paste for v1.
- Persistent whisper groups (already deferred in Phase 4) — not affected by this phase.
- Member roles beyond owner/member (e.g., moderator) — future phase.

</deferred>
